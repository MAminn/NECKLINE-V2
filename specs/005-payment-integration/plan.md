**Branch**: `005-payment-integration` | **Date**: 2026-06-04 | **Spec**: `specs/005-payment-integration/spec.md`

**Input**: Phase 5 of ROADMAP.md — "Wire a real processor into the Phase 4 adapter."

## Summary

Replace the stub payment provider with a real **Paymob** integration behind the existing `PaymentProvider` interface. Paymob is Egypt's leading payment aggregator (cards, wallets, installments, kiosk). The integration uses Paymob's **Intention API** (modern v2 flow): server creates a payment intention → returns a `clientSecret` → frontend redirects to Paymob's hosted checkout page → customer pays → Paymob redirects back + sends webhook → server confirms order.

Key architectural shift: the current checkout flow is **synchronous** (server confirms payment inline). Paymob requires an **async redirect + webhook** flow. We will adapt the flow minimally:
1. `POST /checkout` — creates checkout session (unchanged)
2. `POST /orders` — creates order with `status: 'pending_payment'`, creates Paymob intention, returns `{payUrl}` instead of completed order
3. Frontend redirects to `payUrl`
4. Paymob redirects to `/order-confirmation?orderNumber=...` on success
5. `POST /webhooks/paymob` — receives async confirmation, verifies HMAC signature, confirms order atomically
6. Frontend order-confirmation page polls or shows pending state until webhook confirms

The stub provider remains available via `PAYMENT_PROVIDER=stub` for testing.

## Complexity Tracking

> No constitution violations. Complexity is justified:

| Decision | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| Async redirect + webhook flow (vs synchronous) | Paymob's modern API is intention-based with hosted checkout; no synchronous charge API exists | Keeping synchronous flow would require a different provider (Stripe) — user explicitly chose Paymob |
| Mount webhook route before `express.json()` | HMAC signature verification requires raw request body; global JSON parser destroys raw bytes | Conditional body parsing per-route is fragile and error-prone in Express |
| Stock decrement BEFORE payment intent | Prevents overselling; if Paymob intention fails, we compensate (restore stock) | Decrementing after payment would allow overselling during the payment window |
| `pending_payment` order status | Distinguishes "order created, awaiting payment" from "payment confirmed"; enables polling and reconciliation | Reusing `pending` would conflate "not yet paid" with "payment in flight" |

## Decisions

### Decision 1: Paymob Intention API (hosted checkout) vs iframe/elements

**Rationale**: The Intention API is Paymob's modern v2 flow. It returns a `client_secret` that the frontend uses to redirect to Paymob's fully-hosted checkout page. This gives us **zero PCI scope** (Constitution §4.6) — card data never touches our server or frontend. The alternative (Paymob Elements iframe) requires more frontend complexity and still needs webhook handling.

**Trade-off**: Less control over checkout UI styling. Mitigated by the fact that Paymob's hosted page is professionally designed and localized.

### Decision 2: Stock decrement happens BEFORE creating Paymob intention

**Rationale**: The current `processOrder()` flow already decrements stock atomically before confirming payment. We preserve this: when the customer clicks "Pay Now", we immediately run the DB transaction (decrement stock, create order with `status: 'pending_payment'`, clear cart). Then we create the Paymob intention. If the intention fails, we compensate (restore stock, delete order). This guarantees no overselling.

**Trade-off**: If Paymob is down, the customer sees an error and stock is restored. This is correct behavior — better than selling inventory we can't collect payment for.

### Decision 3: Webhook is the primary confirmation path; redirect is secondary

**Rationale**: Paymob sends a server-to-server webhook when payment succeeds. This is the authoritative signal. The customer's browser redirect back to our site is unreliable (they might close the tab, lose network, etc.). The webhook handler confirms the order atomically. The frontend order-confirmation page shows a "Confirming payment..." state and can poll the order endpoint.

**Trade-off**: Requires the server to be publicly reachable for webhooks. For local development, we use Paymob's test mode + a tool like ngrok, or we simulate webhooks manually.

### Decision 4: Keep stub provider as fallback

**Rationale**: The `PAYMENT_PROVIDER=stub` env var allows running the full checkout flow without Paymob credentials. This is essential for CI, local development, and testing. The factory simply switches between implementations.

**Trade-off**: Slightly more code to maintain. Negligible — the interface is already defined.

## Paymob Integration Details

### Environment Variables (new)

```bash
# Payment provider selection
PAYMENT_PROVIDER=paymob        # 'stub' | 'paymob'

# Paymob credentials (server-side only)
PAYMOB_API_KEY=skl_xxx         # Secret key from Paymob dashboard
PAYMOB_INTEGRATION_ID=xxx      # Integration ID for the payment method
PAYMOB_IFRAME_ID=xxx           # Iframe ID for hosted checkout
PAYMOB_HMAC_SECRET=xxx         # For webhook signature verification
PAYMOB_BASE_URL=https://accept.paymob.com  # or staging URL
```

  async refund(transactionId, amount)

    // Return { success, refundId, amount, currency, status }

### Webhook Flow

```
Paymob → POST /api/v1/webhooks/paymob
  1. Verify HMAC signature (raw body + PAYMOB_HMAC_SECRET)
  2. Parse payload: { type, data: { intention_id, transaction_id, amount, currency, status } }
  3. Deduplicate: check if transaction_id already processed
  4. Find order by intentId (stored in PaymentTransaction.intentId)
  5. Verify amount matches order.total
  6. Atomic transaction:
     - Update Order: status → 'confirmed', paymentStatus → 'succeeded'
     - Update PaymentTransaction: status → 'succeeded', providerTransactionId
  7. Audit event: payment.webhook_confirmed
  8. Return 200 OK
```

### Frontend Flow Changes

```
Current flow:
  Shipping → Review → click Pay Now → POST /orders → wait → redirect to /order-confirmation

New flow:
  Shipping → Review → click Pay Now → POST /orders → receive { order, payUrl }
    → redirect to payUrl (Paymob hosted checkout)
    → Customer pays on Paymob
    → Paymob redirects to /order-confirmation?orderNumber=XXX
    → Frontend shows "Confirming your payment..."
    → Polls GET /orders/:orderNumber every 3s
    → When order.status === 'confirmed', show success screen
    → (Webhook arrives asynchronously in background)
```

3. **Integration tests**: Full flow with `PAYMENT_PROVIDER=stub` (unchanged)

5. **Security tests**: Tampered webhook payload rejected; wrong HMAC rejected; replayed transaction ID rejected

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Paymob webhook not delivered | Frontend polling on order-confirmation page; reconciliation job (Phase 7) |
| Customer closes tab before redirect | Webhook still confirms order; they can look up order later |
| Paymob API downtime | Compensation restores stock; customer sees error; retry allowed |
| HMAC secret leaked | Rotate via env var; webhook endpoint has no auth cookies (stateless) |
| Duplicate webhook delivery | Transaction ID deduplication in webhook handler |
| Amount mismatch (webhook vs order) | Reject webhook with 400; log audit event; manual review |
